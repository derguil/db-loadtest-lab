# 실험 02: 쿼리 최적화(페이지네이션) 전/후

## 목적
목록 조회 API의 쿼리 자체를 개선했을 때, 동일 부하에서 성능이 얼마나 개선되는지 확인한다.

## 변경 내용
- 페이지네이션 없이 전체 조회하던 로직을 page/limit 기반 조회로 변경

## 조건
- 동일 부하 시나리오(max 280 VUs)
- 동일 테스트 시간/도구(k6)

## 결과 요약
- 평균 응답시간이 314.96ms에서 5.73ms로 크게 감소했다.
- 처리량이 175.76 req/s에서 229.61 req/s로 증가했다.
- 결론: pool 튜닝보다 쿼리 최적화의 효과가 훨씬 크다.

## 핵심 수치
| 구분 | 기존 (페이지네이션 없음) | 페이지네이션 추가 후 |
|---|---:|---:|
| 총 요청 수 | 37,037 | 48,399 |
| 처리량 (req/s) | 175.76 | 229.61 |
| 평균 응답시간 (avg) | 314.96ms | 5.73ms |
| p90 | 477.87ms | 10.04ms |
| p95 | 504.63ms | 12.81ms |

## 재현 절차
1. baseline 코드로 테스트 1회 실행
2. 페이지네이션 반영 코드로 테스트 1회 실행
3. 같은 VUs, 같은 duration으로 결과 비교

## 노트
- 커넥션 수를 늘리기 전에 느린 쿼리를 먼저 줄이면, 같은 자원으로 더 큰 개선을 얻을 수 있다.
