# 실험 02: 쿼리 최적화(페이지네이션) 전/후

## 배경

실험 01에서 pool 크기를 늘려도 평소 트래픽에서는 개선 폭이 작다는 걸 확인했다. pool 수치보다 쿼리 자체가 더 큰 영향을 줄 수 있다고 보고, 목록 조회 API의 조회 범위를 줄이는 방향으로 검증해보기로 했다.

## 목표

목록 조회 API의 쿼리 자체를 개선했을 때, 동일 부하에서 성능이 얼마나 개선되는지 확인한다.

## 조건

- 애플리케이션: Nest + Prisma
- DB: MariaDB (local)
- 부하 도구: k6
- 비교 항목: 페이지네이션 없음 vs page/limit 기반 조회
- 부하 강도: 평소(max 280 VUs)

## 변경 내용 / 실험 설계

- 페이지네이션 없이 전체 조회하던 로직을 page/limit 기반 조회로 변경
- 동일 부하 시나리오(max 280 VUs), 동일 테스트 시간/도구(k6)로 전/후 비교

## 결과

### 부하테스트 결과 (k6)

| 구분 | 기존 (페이지네이션 없음) | 페이지네이션 추가 후 |
|---|---:|---:|
| 총 요청 수 | 37,037 | 48,399 |
| 처리량 (req/s) | 175.76 | 229.61 |
| 평균 응답시간 (avg) | 314.96ms | 5.73ms |
| p90 | 477.87ms | 10.04ms |
| p95 | 504.63ms | 12.81ms |

## 결론

- 평균 응답시간이 314.96ms → 5.73ms로 크게 감소했다.
- 처리량이 175.76 → 229.61 req/s로 증가했다.
- pool 튜닝(실험 01)보다 쿼리 최적화의 효과가 훨씬 크다.

## 재현 절차

1. baseline 코드로 테스트 1회 실행
2. 페이지네이션 반영 코드로 테스트 1회 실행
3. 같은 VUs, 같은 duration으로 결과 비교

## 노트

- 데이터베이스 커넥션 풀의 크기를 100 이상으로 증가시켜도 실제 사용되는 커넥션 수는 제한적이었으며, 애플리케이션의 CPU 사용률이 먼저 포화되는 현상이 관찰되었다. 페이지네이션으로 조회 결과의 크기를 약 1/10로 축소하자 CPU 사용률과 pool 사용 개수, 응답 시간이 함께 감소했다.
- 커넥션 수를 늘리기 전에 느린 쿼리를 먼저 줄이면, 같은 자원으로 더 큰 개선을 얻을 수 있다.
